## AMCL 是什么

**AMCL（Adaptive Monte Carlo Localization）** 是 Nav2 里用**静态栅格地图**做定位的模块：在 map 坐标系下估计机器人位姿，并发布 **`map → odom`** 的 TF，用来修正里程计漂移。

需要特别强调的一点：

- **它是定位，不是建图。**  
  SLAM / `slam_toolbox` / gmapping 一类负责「边走边画地图」；AMCL 假设**地图已经建好**，只在已知地图上回答「我在哪」。 建图与定位是两件不同的事，不要混在一个节点里理解。

在系统里，它通常与 `map_server` 一起由 `nav2_bringup` 的 `localization_launch` 拉起，节点名一般为 `amcl`。


## 算法在思想上是什么

### 粒子滤波：用很多「可能」逼近一个「真实」

现实世界充满不确定性：里程计会滑、激光有噪声、地图也不完美。因此不能轻易的得到真实的结果。

粒子滤波的做法是——用**大量粒子**表示「机器人可能在哪」的分布；**每个粒子是一次采样**，粒子越多，对分布的刻画越细，月逼近真实的结果，但算力也越高。

同时，真实世界还有一个更微妙的点：**当事件已经发生，其可能性其实已经坍塌了，只是我们不知道而已**

所以需要传感器量测：用当前激光与地图比对，看哪些粒子更「像真的」——每个粒子带有 **weight（权重）**，权重越大，表示该位姿假设与观测越一致。

### 重采样与 Adaptive

针对两件事，工程上一定会做 **重采样**：

1. **算力**：不能无限增加粒子，要用高权粒子替换低权粒子，把计算集中在「还有希望」的区域；
2. **确定性**：观测不断进来，低权粒子代表的可能性可以淘汰。

**AMCL** 就是在 2D 激光 + 栅格地图场景下对上述粒子滤波的具体实现，Nav2 里默认激光模型是 **likelihood_field（似然场）**。

它相比「教科书粒子滤波」多出的自适应主要体现在：

| 机制 | 作用 |
|------|------|
| **KLD 采样** | 根据粒子空间占用（kd-tree 叶节点数）决定本轮需要多少粒子，而不是固定死个数 |
| **分簇统计** | 粒子按位姿聚类后，取**权重最大的簇**的均值、协方差，输出一个 `/amcl_pose`，而不是随便挑一个粒子 |
| **w_slow / w_fast** | 观测持续很差时，重采样阶段按比例**全局随机撒点**，防止整团粒子跟丢 |

### 预测与更新

粒子滤波是贝叶斯滤波的一种蒙特卡洛实现，一帧里仍然可以记成两大步：

| 步骤 | 在 AMCL 里谁来做 | 改什么                             |
|------|------------------|---------------------------------|
| **预测** | `MotionModel::odometryUpdate` | 粒子的 **pose**（根据 odom 增量 + 高斯噪声） |
| **更新** | `Laser::sensorUpdate` → `pf_update_sensor` | 粒子的 **weight**（激光 + 地图）         |

之后还有 **重采样**（`pf_update_resample`）和 **输出**（`getMaxWeightHyp` → 位姿与 TF）。

## 代码结构：包内大致分几块

`nav2_amcl` 不是单文件算法，而是「ROS 壳 + C 粒子滤波 + 传感器/地图」的分层：

| 目录 / 文件 | 职责 |
|-------------|------|
| **`src/amcl_node.cpp`** | 主节点 `AmclNode`：订阅发布、lifecycle、**`laserReceived` 总调度** |
| **`src/pf/`** | 粒子滤波核心（`pf.c`）：采样、权重归一化、重采样、KLD、聚类 |
| **`src/map/`** | 栅格地图 `map_t`、占据栅格、距离场 `map_update_cspace` |
| **`src/sensors/laser/`** | 激光观测模型（似然场、beam 等），继承 `Laser` |
| **`src/motion_model/`** | 运动模型插件（差速、全向），预测阶段改 pose |
| **`src/main.cpp`** | 只创建 `AmclNode` 并 `spin` |

进程入口极简，没有第二个「定位线程」：

```c++
int main(int argc, char ** argv)
{
  rclcpp::init(argc, argv);
  auto node = std::make_shared<nav2_amcl::AmclNode>();
  rclcpp::spin(node->get_node_base_interface());
  rclcpp::shutdown();
```

**真正的滤波循环在激光回调里同步跑完**；`spin` 负责把 map、初值等其它回调也驱动起来。


## ROS 层：订阅、发布与初始化

### 关键成员：数据从哪进、从哪出

`AmclNode` 继承 `nav2_util::LifecycleNode`，通过节点的状态变化完成自身状态的切换，和「定位数据」直接相关的成员可以分成以下几个部分（声明见 `amcl_node.hpp`）：

```text
地图：     map_sub_  → mapReceived
初值：     initial_pose_sub_  → initialPoseReceived
激光：     laser_scan_sub_ + laser_scan_filter_  → laserReceived
里程计：   不订阅话题，用 tf_buffer_ 查 odom → base 的 TF
输出：     pose_pub_（amcl_pose）、particle_cloud_pub_、tf_broadcaster_
```

为了保证**这一帧 scan 在 odom 坐标系下可对齐**之后，再进 `laserReceived`，避免 TF 未到就算滤波，激光同时使用`message_filters` + `tf2_ros::MessageFilter`：。

### `on_configure`：按固定顺序搭架子

节点不会一启动就滤波。`on_configure` 里按依赖顺序初始化（节选）：

```C++
AmclNode::on_configure(const rclcpp_lifecycle::State & /*state*/)
{
  RCLCPP_INFO(get_logger(), "Configuring");
  callback_group_ = create_callback_group(
    rclcpp::CallbackGroupType::MutuallyExclusive, false);
  initParameters();
  initTransforms();
  initParticleFilter();
  initLaserScan();
  initMessageFilters();
  initPubSub();
  initServices();
  initOdometry();
  executor_ = std::make_shared<rclcpp::executors::SingleThreadedExecutor>();
  executor_->add_callback_group(callback_group_, get_node_base_interface());
  executor_thread_ = std::make_unique<nav2_util::NodeThread>(executor_);
  return nav2_util::CallbackReturn::SUCCESS;
}
```

要点：

- **`initParticleFilter`**：先 `pf_alloc`，此时还没有有效 `map_` 上的距离场；
- **`initMessageFilters`**：把 **`laserReceived`** 注册到激光专用 callback group；
- **`initPubSub`**：地图、初值订阅，以及 lifecycle 发布者；
- **末尾 `NodeThread`**：单独 executor 跑激光回调，避免和 TF 定时器互卡。

`initTransforms` 创建 `tf_buffer_` / `tf_listener_` / `tf_broadcaster_`；

`initMessageFilters` 与 `initPubSub` 的代码逻辑是：**scan 走滤波线程，map 与 initialpose 走默认回调组**。

### `on_activate`：打开「闸门」

configure 之后 `active_` 仍为 false，激光回调会直接 return。`on_activate` 里激活发布者并置位：

```c++
AmclNode::on_activate(const rclcpp_lifecycle::State & /*state*/)
{
  RCLCPP_INFO(get_logger(), "Activating");
  pose_pub_->on_activate();
  particle_cloud_pub_->on_activate();
  first_pose_sent_ = false;
  active_ = true;
```

Nav2 里通常由 **lifecycle_manager** 自动 configure / activate；单独调试时才需要手敲 `ros2 lifecycle set`。


## 另外两条数据线（进 `laserReceived` 之前）

定位是**数据驱动**的，除了激光，还有两条线必须先就绪。

### 静态地图 `/map` → `map_`

```c++
void AmclNode::mapReceived(const nav_msgs::msg::OccupancyGrid::SharedPtr msg)
{
  ...
  handleMapMessage(*msg);
  first_map_received_ = true;
}

void AmclNode::handleMapMessage(const nav_msgs::msg::OccupancyGrid & msg)
{
  std::lock_guard<std::recursive_mutex> cfl(mutex_);
  ...
  freeMapDependentMemory();
  map_ = convertMap(msg);
```

- `convertMap`：ROS 栅格 → C 结构 `map_t`；
- **`freeMapDependentMemory` 会清空 `lasers_`**：换图后旧的 `LikelihoodFieldModel` 不能继续用旧 map 指针；
- 距离场 **`occ_dist`** 在**创建 `Laser` 对象**时由 `map_update_cspace` 建立，不在 `mapReceived` 里做。

没有 `first_map_received_`，`laserReceived` 会打 `Waiting for map....` 并 return。

### 初始位姿 `/initialpose` → `pf_init`

用户通过 RViz「2D Pose Estimate」或话题 `/initialpose` 提交估计。`handleInitialPose` 在 map 帧下重撒粒子：

```C++
  pf_init(pf_, pf_init_pose_mean, pf_init_pose_cov);
  pf_init_ = false;
  ...
  initial_pose_is_known_ = true;
```

没有已知初值时，`publishAmclPose` / `publishParticleCloud` 会直接 return——**有 scan 也可能没有位姿输出**。

## 主路径：`laserReceived` 一帧怎么走

实现定位**主要的入口就是 `AmclNode::laserReceived`**。下面按执行顺序拆（与源码一致）。

### 就绪检查

```c++
void AmclNode::laserReceived(sensor_msgs::msg::LaserScan::ConstSharedPtr laser_scan)
{
  std::lock_guard<std::recursive_mutex> cfl(mutex_);

  if (!active_) {return;}
  if (!first_map_received_) {
    if (checkElapsedTime(2s, last_time_printed_msg_)) {
      RCLCPP_WARN(get_logger(), "Waiting for map....");
      last_time_printed_msg_ = now();
    }
    return;
  }
```

未 activate、没有地图，都不继续。整函数在 **`mutex_`** 下执行，与 `mapReceived`、`handleInitialPose` 互斥。

### 激光模型对象：第一次见到该 `frame_id`

```639:646:nav2_amcl/src/amcl_node.cpp
  if (frame_to_laser_.find(laser_scan_frame_id) == frame_to_laser_.end()) {
    if (!addNewScanner(laser_index, laser_scan, laser_scan_frame_id, laser_pose)) {
      return;  // could not find transform
    }
  } else {
    laser_index = frame_to_laser_[laser_scan->header.frame_id];
  }
```

`addNewScanner` 里 `createLaserObject()` 按参数创建 `LikelihoodFieldModel` 等，**此时才绑定当前 `map_`**。

所以顺序必须是：**先有 map，再有 Laser 实例**。

### 里程计：查 TF，算增量 `delta`

```c++
  if (!getOdomPose(
      latest_odom_pose_, pose.v[0], pose.v[1], pose.v[2],
      laser_scan->header.stamp, base_frame_id_))
  {
    RCLCPP_ERROR(get_logger(), "Couldn't determine robot's pose associated with laser scan");
    return;
  }
```

注意：AMCL **不订阅 `/odom` 话题**，而是在 `getOdomPose` 里查 **TF（odom → base）**。

与上次保存的 `pf_odom_pose_` 做差得到 **`delta`**，供运动模型做预测。

是否在本帧做观测更新，由 `shouldUpdateFilter` 根据 `update_min_d`、`update_min_a` 判断；机器人几乎不动时可以跳过更新，这是省电策略，不是故障。

### 预测：只改 pose

```c++
    if (lasers_update_[laser_index]) {
      motion_model_->odometryUpdate(pf_, pose, delta);
    }
```

差速模型在 `differential_motion_model.cpp` 里把 `delta` 拆成旋转-平移-旋转，对**每个粒子**加噪声更新位姿，**不动 weight**。

### 更新：只改 weight

```C++
  if (lasers_update_[laser_index]) {
    updateFilter(laser_index, laser_scan, pose);
```

`updateFilter` 把 `LaserScan` 转成 `LaserData`（距离 + bearing），再调用：

```C++
  lasers_[laser_index]->sensorUpdate(pf_, reinterpret_cast<nav2_amcl::LaserData *>(&ldata));
  lasers_update_[laser_index] = false;
  pf_odom_pose_ = pose;
```

似然场模型里，`sensorUpdate` 最终进 `pf_update_sensor`，对每个粒子、每条降采样光束查 **`occ_dist`**（光束终点离障碍越近，权重乘性越大），再归一化并更新 `w_slow` / `w_fast`。

### 重采样：复制高权、必要时全局恢复

```C++
    if (!(++resample_count_ % resample_interval_)) {
      pf_update_resample(pf_, reinterpret_cast<void *>(map_));
      resampled = true;
    }
```

`pf_update_resample`（`pf.c`）里：双缓冲集合切换、按权重轮盘赌复制 pose、按 KLD 决定粒子数、按 `w_diff` 决定是否调用 `random_pose_fn(map_)` 撒随机粒子。

`pf` 里维护 **两个 sample set**（`sets[0]` / `sets[1]`），就是为了重采样时读写分离；`current_set` 指向当前在用的一套。

### 输出：粒子云、位姿、`map → odom`

```c++
    if (!force_update_) {
      publishParticleCloud(set);
    }
  }
  if (resampled || force_publication || !first_pose_sent_) {
    ...
    if (getMaxWeightHyp(hyps, max_weight_hyps, max_weight_hyp)) {
      publishAmclPose(laser_scan, hyps, max_weight_hyp);
      calculateMaptoOdomTransform(laser_scan, hyps, max_weight_hyp);
      ...
        sendMapToOdomTransform(transform_expiration);
```

- **`publishParticleCloud`**：把当前粒子集发到 `/particle_cloud`；
- **`getMaxWeightHyp`**：对所有簇按权重排序，取**最大权簇**的均值和协方差；
- **`calculateMaptoOdomTransform` + `sendMapToOdomTransform`**：发布 **map → odom**，把里程计系「拧」到与地图一致。

若本帧未重采样且已有有效 TF，末尾还可能**重复发送上一次 map→odom**，避免 TF 树断档。

## 把一帧串成一张表

| 顺序 | 代码位置 | 粒子滤波语义 |
|------|----------|----------------|
| 1 | 门闩 `active_` / `first_map_received_` | 数据未就绪不算 |
| 2 | `getOdomPose` + `shouldUpdateFilter` | 准备 `delta` |
| 3 | `motion_model_->odometryUpdate` | **预测** → pose |
| 4 | `updateFilter` → `sensorUpdate` | **更新** → weight |
| 5 | `pf_update_resample` | 重采样 + KLD + 恢复 |
| 6 | `publishParticleCloud` / `publishAmclPose` / TF | 输出 |

## 小结

- **AMCL** = 已知地图上的蒙特卡洛定位，不是建图。
- **思想** = 粒子表分布，预测改 pose、观测改 weight，重采样 + KLD + 分簇输出。
- **代码** = `AmclNode` 接线，`pf` / `map` / `Laser` / `MotionModel` 分工；**一帧的总线在 `laserReceived`**。
