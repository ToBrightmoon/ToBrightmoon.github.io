## 前言

很多 AMCL 教程都会说：

> AMCL 用于机器人定位。

但很少继续说明：**定位什么？**

更准确一点说：

* 粒子滤波器内部维护的状态量是什么？
* 这个状态量属于哪个坐标系？
* 激光数据进入滤波器后，又是如何与这个状态量发生联系的？

这些问题如果不弄清楚，后面阅读运动模型、观测模型、重采样时都会比较混乱。

因此这篇文章只回答一个问题：

> AMCL 到底在估计谁的位姿？


## 手册上的标准答案


首先，在 [ROS 1 Wiki / Nav2 AMCL 官方文档](https://wiki.ros.org/amcl) 中，关于坐标系发布（Published Transforms）的定义非常明确：

* **AMCL 核心估计的，就是 `base_link`（机器人机体）在 `map`（静态全局地图）坐标系下的位姿变换。**

![ros_wiki_png](/images/amcl/01/amcl_wiki_frame.png)

在工程实现上，为了避免与里程计节点发布的 `odom → base_link` 产生双重父节点的冲突，AMCL 采用了一种间接修正的技巧：它不直接发布 `map → base_link`，而是通过查阅里程计的 TF，发布 **`map → odom`** 的变换。


这就引出了我们在源码中必须死盯的三个基本参数。

在 `AmclNode` 构造函数和 `initParameters` 回调中，我们可以看到这三个参数的作用与意义：

```c++
// nav2_amcl/src/amcl_node.cpp
get_parameter("global_frame_id", global_frame_id_); // 默认: "map" (全局静态世界)
get_parameter("base_frame_id", base_frame_id_);     // 默认: "base_footprint" (机器人底盘投影)
get_parameter("odom_frame_id", odom_frame_id_);     // 默认: "odom" (存在累积漂移的里程计坐标系)

```

后续主要的位姿变化也是在这三个位姿坐标系展开。

## AMCL中维护的状态

想要知道真正估计的状态，知道我们估计的结果，可以从粒子结构体里。

在 `pf.hpp` 中：

```cpp
typedef struct
{
  pf_vector_t pose;
  double weight;
} pf_sample_t;
```

一个粒子只有两个核心成员：`pose` 与 `weight`

其中：`weight`表示当前粒子的可信度。而`pose`才是滤波器真正维护的状态。

也就是说：

> AMCL 本质上是在维护大量 pose 假设。

粒子云中的每一个粒子，都代表机器人可能存在的位置。

## 初始第一帧位姿

AMCL 是一套蒙特卡洛（粒子滤波）方法，任何迭代算法都需要一个初始值作为起点，amcl也不例外。

在AMCL的源码，通过 `on_configure`，完成了初始信息的回调函数的注册

```c++
void AmclNode::initPubSub()
{
  RCLCPP_INFO(get_logger(), "initPubSub");

  ......

  initial_pose_sub_ = create_subscription<geometry_msgs::msg::PoseWithCovarianceStamped>(
    "initialpose", rclcpp::SystemDefaultsQoS(),
    std::bind(&AmclNode::initialPoseReceived, this, std::placeholders::_1));
  
  ......

  RCLCPP_INFO(get_logger(), "Subscribed to map topic.");
}
```

用户在 RViz 中点击 `2D Pose Estimate` 或向 `/initialpose` 话题发出的消息，进入`AmclNode::initialPoseReceived`中

### 初值的校验

代码在使用初值的时候，会进行数值与坐标系的校验，保证数值与坐标系的合规性。

```c++
if (!nav2_util::validateMsg(*msg)) { return; } // 位姿本身不能含有 NaN 损坏数据
if (nav2_util::strip_leading_slash(msg->header.frame_id) != global_frame_id_) {
  // 拦截：传入的初值必须是在 global_frame_id_（即 map）下的绝对位姿！
  return; 
}


### 消除传输延迟的底盘补偿

同时，为了补偿，在你发送消息与收到消息之间可能因为机器人运动带来的位姿变化，`handleInitialPose` 内部做了一次巧妙的 **时间戳回溯补偿** ：

```c++
// 1. 查找从 initialpose 的历史时间戳到 AMCL 当前时刻，底盘自己走了多少相对运动
tx_odom = tf_buffer_->lookupTransform(base_frame_id_, tf2_ros::fromMsg(msg.header.stamp),
                                       base_frame_id_, tf2_time, odom_frame_id_);
// 2. 将用户指定的旧图上位姿，乘上这段时间的里程计运动增量，更新为当前时刻的最准位姿
tf2::Transform pose_new = pose_old * tx_odom_tf2;

```

### 滤波器的几何底色：全员 `map` 系

计算完成后，初值通过 `pf_init` 注入粒子滤波器：

```c++
pf_init_pose_mean.v[0] = pose_new.getOrigin().x();
pf_init_pose_mean.v[1] = pose_new.getOrigin().y();
pf_init_pose_mean.v[2] = tf2::getYaw(pose_new.getRotation());

pf_init(pf_, pf_init_pose_mean, pf_init_pose_cov);

```

### 结论的确认

但是现在仅仅能确认的是，位姿是在map坐标系下的，不能确认到底是哪一个传感器在map下的坐标系，所以继续从源码出发。

在处理初始位姿时：

```c++
tx_odom = tf_buffer_->lookupTransform(
    base_frame_id_,
    ...,
    base_frame_id_,
    ...,
    odom_frame_id_);
```

这里出现的始终是 `base_frame_id_`,整个初始化过程都没有出现 `laser`,并且激光的外参也没有注册完成，所以sample->pose不可能表示激光位姿。它表示的只能是：T_map_base，既机器人底盘在map中的位姿。


> **结论**：由此开始，AMCL 底层滤波器核心 `pf_t` 内部维护的所有粒子 `sample->pose`，其内在的数学语义**全部在 `map` 坐标系下**。

它们代表了机器人在静态地图中的成百上千种可能的位置假设。

## 传感器数据流：局部雷达到全局地图的投影链条

明确了粒子全都在 `map` 系下之后，我们来看一帧局部的激光雷达点云（在 `laser_frame` 下）进来时，是如何与地图产生碰撞并完成闭环的。

###  动态注册外参

在 `laserReceived` 主数据总线中，AMCL 支持多雷达输入。

首先判断，这个激光传感器是不是出现过，没有出现就调用`addNewScanner`,否则就直接得到某个某个激光器的索引，方便后续查询

```c++
 ......

  std::string laser_scan_frame_id = nav2_util::strip_leading_slash(laser_scan->header.frame_id);
  last_laser_received_ts_ = now();
  int laser_index = -1;
  geometry_msgs::msg::PoseStamped laser_pose;

  // Do we have the base->base_laser Tx yet?
  if (frame_to_laser_.find(laser_scan_frame_id) == frame_to_laser_.end()) {
    if (!addNewScanner(laser_index, laser_scan, laser_scan_frame_id, laser_pose)) {
      return;  // could not find transform
    }
  } else {
    // we have the laser pose, retrieve laser index
    laser_index = frame_to_laser_[laser_scan->header.frame_id];
  }
  ......

```

然后在 `addNewScanner` 动态测算外参：

```c++
// 查询当前雷达 frame 到 base_frame_id_ 的静态 TF 外参
tf_buffer_->transform(ident, laser_pose, base_frame_id_, transform_tolerance_);

pf_vector_t laser_pose_v;
laser_pose_v.v[0] = laser_pose.pose.position.x; // 雷达相对于底盘中心的 X 轴偏移 (米)
laser_pose_v.v[1] = laser_pose.pose.position.y; // 雷达相对于底盘中心的 Y 轴偏移 (米)
laser_pose_v.v[2] = 0; // 航向角初置 0，后续计算

lasers_[laser_index]->SetLaserPose(laser_pose_v); // 将该外参牢牢绑定在对应的 LaserModel 实例中

```

> **结论**：`lasers_[laser_index]->laser_pose_` 存储的是 laser to base 的静态外参**。它只取决于硬件安装位置，完全不随机器人的运动而改变。

### 似然场的坐标转换

当数据经过校验，完成运动更新之后，`updateFilter` 最终会将雷达降采样数据灌入 `LikelihoodFieldModel::sensorFunction` 中。

对于**每一个粒子**，代码将进行数据转换，完成数据从激光系到map系的转换：


```c++
for (j = 0; j < set->sample_count; j++) {
    sample = set->samples + j;
    pose = sample->pose; // 第一步：拿到当前粒子在 map 下的【底盘假设位姿】 (X_map, Y_map, Yaw_map)

    // 第二步：复合静态外参！将该假设位姿，变换为该粒子视角下，【雷达在 map 下的假设位姿】
    pose = pf_vector_coord_add(self->laser_pose_, pose); 

    for (i = 0; i < data->range_count; i += step) {
        double obs_range = data->ranges[i][0];     // 拿到当前激光束的局部测量距离值 (局部米)
        double obs_bearing = data->ranges[i][1];   // 拿到当前激光束在雷达极坐标系下的夹角 (局部弧度)

        // 第三步：利用三角函数，将局部测距终点（Endpoint）直接投影至地图【map 坐标系】下
        hit.v[0] = pose.v[0] + obs_range * cos(pose.v[2] + obs_bearing);
        hit.v[1] = pose.v[1] + obs_range * sin(pose.v[2] + obs_bearing);

        // 第四步：拿这个 map 系下的终点坐标 hit.v，去查 map_t 栅格地图里的边界距离场 (occ_dist)
        // 离障碍物越近，该粒子的生存权重 (weight) 乘性越高！不动 pose，只改 weight！
    }
}

```

综上，整个雷达点云的处理就是，当第一帧出现时计算外参，当更新时利用外参得到激光点云在全局坐标系下的位置。

## 五、 总结

这一篇回答了一个的问题：

> AMCL 到底在估计什么？

从源码可以看到：

* 粒子内部维护的是 `sample->pose`
* `sample->pose` 从初始化开始就在 `map` 坐标系下
* 它描述的是机器人底盘 `base`
* 激光坐标系只是观测阶段通过外参投影到地图
* 整个滤波器真正估计的状态始终是：

> T_map_base
即：
> 机器人在地图中的位姿
