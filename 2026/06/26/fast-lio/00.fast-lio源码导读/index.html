Fast-lio源码阅读笔记

发表于2026-06-27|更新于2026-06-27|状态估计激光SLAM

基于fast-lio2代码ros2分支fast-lio2阅读整理，与论文 FAST-LIO2 (arXiv:2107.06829) 及官方流程图对齐。

本文为个人读码笔记，不是官方文档或入门教程；理解有误欢迎指正。

背景与定位

FAST-LIO 是香港科技大学 MARS 实验室开源的 LiDAR-惯性里程计（LIO）方案,全称为 Fast Direct LiDAR-inertial Odometry。

Direct是什么意思

相对 LOAM、LeGO-LOAM 等基于曲率提取边点/面点的方法，FAST-LIO2 不再做特征提取，而是对降采样后的原始点云直接做 scan-to-map 的点到面匹配，并在 IKFoM 流形 ESKF 中与 IMU 紧耦合。

定位 vs 建图

SLAM 可粗分为：

  • 定位（Localization）：估计载体位姿、速度、传感器 bias 等
  • 建图（Mapping）：维护环境地图供匹配

FAST-LIO 是 Odometry（里程计），不做闭环。定位与建图在同一个滤波器里紧耦合完成，而不是两个独立线程各跑各的。

为什么要读「数据处理」

算法论文描述的是理想模型；落地时必须处理：时间同步、运动畸变、buffer 边界、静止初始化、地图有界维护等。FAST-LIO 的主干逻辑清晰，工程细节集中在 laserMapping.cpp 与 IMU_Processing.hpp。

算法概述

一帧 LiDAR 的处理可概括为：

IMU 预测

用 IMU 积分推进状态 x 与协方差 P，得到 LiDAR 更新前的先验。

运动畸变校正

LiDAR 一帧扫描持续数十 ms，期间载体在动。各点在不同时刻采集，若不做补偿，几何会「糊」。
FAST-LIO 用 IMU 轨迹把每个点变换到扫描结束时刻 $t_{\mathrm{end}}$ 的 LiDAR 系。

IEKF 更新

对每个（降采样）点：

  1. 变换到世界系
  2. 在 ikd-tree 地图中 kNN，拟合局部平面
  3. 计算点到面残差与 12 维雅可比
  4. 迭代修正状态直至收敛

增量建图

用 update 之后的状态把点投到世界系，按体素策略增量写入 ikd-tree。

代码总览

与论文流程图的关系

官方 pipeline：

fast-lio的源码实现与论文也基本一致，只是增加了一些处理细节。

主入口：timer_callback

文件：src/laserMapping.cpp
函数：LaserMappingNode::timer_callback()

100 Hz 定时器触发；仅当 sync_packages() 成功组装一帧时才执行主流程：

步骤 函数 / 模块 说明
1 sync_packages 从 buffer 组装一帧 LiDAR + IMU
2 p_imu->Process IMU 初始化 / predict + 去畸变
3 lasermap_fov_segment 局部地图立方体滑窗 + 盒删除
4 体素降采样 downSizeFilterSurf → feats_down_body
5 kf.update_iterated_dyn_share_modified IEKF（内部回调 h_share_model）
6 publish_odometry 等 发布里程计、点云、path
7 map_incremental 增量更新 ikd-tree

线程分工

  • 传感器回调（livox_pcl_cbk / imu_cbk）：预处理 + 入队，轻量，避免阻塞
  • 定时器线程（timer_callback）：积分、滤波、匹配、建图，重计算

数据处理（回调层）

LiDAR 回调

文件：src/laserMapping.cpp · livox_pcl_cbk()

主要职责：

  1. 时间单调性检查：若 cur_time < last_timestamp_lidar，清空 lidar_buffer（rosbag 回环/驱动重启）
  2. 可选软同步：time_sync_en 时用首批 LiDAR/IMU 估固定时差 timediff_lidar_wrt_imu
  3. 预处理：p_pre->process() → 统一为 PointCloudXYZI，盲区过滤，点时间写入 curvature（ms）
  4. 入队：lidar_buffer + time_buffer，不做 EKF / kNN

IMU 回调

文件：src/laserMapping.cpp · imu_cbk()

主要职责：

  1. 时间轴对齐 LiDAR：
    • 配置偏移：stamp -= time_diff_lidar_to_imu（约定 lidar_time - imu_time）
    • 软同步优先：time_sync_en 时用 timediff_lidar_wrt_imu 修正
  2. 时间回退保护：timestamp < last_timestamp_imu 时清空 imu_buffer
  3. 入队：imu_buffer.push_back；积分在 ImuProcess::Process 中进行

注：源码里 IMU 回退时的日志字符串误写为 "lidar loop back"，实为 IMU 时间回退。

以 LiDAR 时间为传感器主时间轴；IMU stamp 被平移到该时间系。
一帧的扫描结束时刻由末点 curvature / 1000（秒）+ 帧头时间得到，见 sync_packages。

数据同步

定时器

文件：src/laserMapping.cpp · LaserMappingNode 构造函数

100 Hz 尝试同步并处理；并非每 tick 都有一帧 LiDAR。

sync_packages 逻辑

文件：src/laserMapping.cpp · sync_packages()

成功条件：

  1. lidar_buffer 与 imu_buffer 均非空
  2. IMU 最新时间 ≥ lidar_end_time（覆盖完整扫描区间）

关键机制：

  • lidar_pushed 标志：已取出 LiDAR 但 IMU 尚未够时 不能 pop LiDAR，下次继续同步同一帧
  • **lidar_end_time**：由末点 curvature 估计；异常时用历史平均扫描周期兜底
  • IMU 截取：弹出 stamp < lidar_end_time 的全部 IMU 放入 meas.imu
  • 帧间 IMU 衔接：在 UndistortPcl 里 v_imu.push_front(last_imu_)，保证积分连续（不在 sync_packages 内完成）

去畸变与状态预测

文件：src/IMU_Processing.hpp

Process 分支

静止初始化 IMU_init

前提：启动后约 MAX_INI_COUNT = 10 帧内设备近似静止。

量 初始化方式
bg mean(gyro)
grav -mean(acc) 方向，映射到 S² 流形
外参 R_IL, t_IL 配置文件 / 标定
ba 不由均值直接赋值，靠后续 EKF 在线估计

初始化完成前不会进入去畸变与 LiDAR 更新。

UndistortPcl 五步

  1. 插入 last_imu_ 到当前帧 IMU 序列头
  2. 按 curvature 对点云排序
  3. 前向 IMU 积分：每段 kf.predict(dt, Q, in)，记录 IMUpose 轨迹
  4. 推到 pcl_end_time（扫描末时刻，LiDAR 更新先验必须在此刻）
  5. 反向去畸变：从扫描末向前，将 $t_i$ 时刻点变换到 $t_{\mathrm{end}}$ 的 LiDAR 系

去畸变与 predict 在同一函数内完成，共用 IMU 积分结果。

变换链（概念上）：

输出：feats_undistort（body/LiDAR 系，时刻统一到 $t_{\mathrm{end}}$）。

局部地图滑窗

文件：src/laserMapping.cpp · lasermap_fov_segment()

做什么

以 LiDAR 当前位置为中心，维护边长 cube_len 的局部立方体地图。当雷达靠近立方体边界时：

  1. 平移窗口
  2. 对移出区域的点做 ikd-tree 盒删除（Delete_Point_Boxes）

不是什么

不是对每次 kNN 查询做 FOV 锥形裁剪。
kNN 在当前 ikd-tree（已是滑窗后的局部地图）上搜索；滑窗保证地图有界、搜索量可控。

在 pipeline 中的位置

在 降采样之前、IEKF 之前 调用，先删远点再匹配。

IEKF 状态更新

观测模型 h_share_model

文件：src/laserMapping.cpp · h_share_model()

对每个降采样点：

  1. 世界系变换：p_W = R · (R_IL · p_L + t_IL) + p
  2. 条件 kNN：仅当 ekfom_data.converge == true 时重新 Nearest_Search（见 8.3）
  3. 平面拟合：5 近邻 + esti_plane，阈值 0.1 m
  4. 残差：r = nT · p_W + d，鲁棒门限筛选
  5. 雅可比：h_x 为 m×12，对应 [pos, rot, ext_R, ext_T]
  6. 测量向量：h(i) = -r（配合 IKFoM innovation 定义）

速度、bias、重力不进入 12 列雅可比，但可通过协方差 P 交叉项被间接修正。

迭代更新 update_iterated_dyn_share_modified

文件：include/IKFoM_toolkit/esekfom/esekfom.hpp

在 IMU 传播点 x_prop 处迭代（默认 4 次）：

特征点多时用信息形式（HT · H 仅 12×12），避免对 m×m 求逆，保证实时性。

迭代与重匹配

并非每一轮迭代都重新 kNN：

  • 初始 converge = true，首轮会搜索
  • 若某轮更新量 $|\delta x_i|$ 仍大于 limit[i]，置 converge = false，下轮复用已有近邻，节省计算
  • 当连续 2 次满足收敛条件，更新协方差 P 并退出

首帧建树（尚无地图可匹配）

ikdtree.Root_Node == nullptr 时：

  • 将降采样点投世界系
  • ikdtree.Build(...)
  • 直接 return，不做 IEKF

地图增量更新

文件：src/laserMapping.cpp · map_incremental()

顺序约束

必须在 IEKF update 之后调用。
若用 predict 状态入图，会把本帧匹配误差固化进地图。

近邻来源

map_incremental 不重新 kNN，复用 IEKF 阶段 h_share_model 已算好的 Nearest_Points[i]。

体素策略

对每个点（世界系）：

情况 处理
无近邻或 EKF 未初始化 加入 PointToAdd（带 ikd-tree 内部降采样）
最近地图点不在当前体素内 加入 PointNoNeedDownsample（直接插入）
体素内已有更近代表点 跳过当前点
当前点更靠近体素中心 加入 PointToAdd（降采样更新）

体素边长：filter_size_map_min。

启动时序与边界条件

帧级时序

阶段 条件 行为
第 1 帧 LiDAR flg_first_scan 仅设 first_lidar_time，return
IMU 初始化期 imu_need_init_ 只做 IMU_init，return
首棵地图树 ikdtree.Root_Node == nullptr Build 后 return，无 IEKF
正常帧 以上均否 完整 pipeline

其他边界

  • INIT_TIME = 0.1 s：flg_EKF_inited 为 false 时，影响 map_incremental 体素逻辑
  • 点数过少：feats_down_size < 5 跳过本帧
  • 无有效匹配点：h_share_model 置 valid = false
  • 时间回退：LiDAR/IMU buffer 清空，防止错误同步
  • **extrinsic_est_en**：为 false 时外参雅可比列置零
